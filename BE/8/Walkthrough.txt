Challenge Document (Player-Facing)
Title: The Operator's Block
Category: Pwn / Binary Exploitation
Difficulty: Easy

Briefing:
You've gained access to a legacy authentication console for a local block-building server. It restricts operator (OP) privileges to specific usernames.

However, the server runs on outdated, unsafe C functions. Find a way to exceed the allocated character limit and overwrite the underlying game mode memory state to grant yourself CREATIVE mode.

Rules:

Run the binary locally on a Linux machine.

Trigger the hidden drop_seed() function to get the flag.

Organizer Walkthrough (Intended Solution)
This is a classic "hello world" of binary exploitation. The vulnerability is the gets() function, which does not check the length of the user's input.

Decompilation / Analysis: A player opens the binary or reads the prompts and notices they are asked for an IGN (In-Game Name). If they throw it into a decompiler, they will see that player_name has a 32-byte capacity, and right next to it is the is_admin variable.

The Flaw: Because gets() will keep reading input until the user presses Enter, a user can type more than 32 characters.

The Overflow: Once the 32nd character is written, the 33rd character spills over into the adjacent memory space—which is occupied by ctx.is_admin.

The Exploit: * The player runs the binary.

When prompted for the IGN, they type 33 characters (e.g., AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA).

The first 32 As fill the player_name buffer.

The 33rd A overwrites ctx.is_admin changing its value from 0 to 65 (the ASCII value of 'A').

Execution: Because is_admin is now 65 (which is != 0), the check if (ctx.is_admin != 0) passes, the drop_seed() function runs, and the flag decrypts: CH4KR4X2{m1n3_b1n_h4ck_P0D5}.
